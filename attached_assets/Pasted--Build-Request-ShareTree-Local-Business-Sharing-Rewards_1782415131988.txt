# Build Request: ShareTree — Local Business Sharing & Rewards Platform

Build a polished, mobile-first web application named **ShareTree**.

## Product purpose

ShareTree connects local businesses with everyday people who want to earn rewards by sharing approved business campaigns with their real networks.

Businesses create campaigns to promote a website, social profile, product, service, event, or physical location. Users discover locally relevant campaigns, share them through approved channels, and earn rewards only when the system verifies eligible campaign activity.

The platform must feel trustworthy, modern, easy to use, and suitable for users across the United States.

# Core user types

## 1. Free User / Sharer

Users sign up for free using Google OAuth.

On first sign-in, automatically create a user profile with:

* Full name
* Email address
* Google profile photo
* Unique user ID
* Referral code
* Account creation date
* Default location permission prompt
* City, state, ZIP code, and service-area preferences
* Earnings balance
* Pending earnings balance
* Lifetime earnings
* Share history
* Click and conversion history
* Payout status
* Verification / compliance status

Users can:

* Browse recommended local and national campaigns
* Share approved campaign links through supported share methods
* Copy a unique tracking link
* Use native mobile sharing where available
* Share by text message, email, Facebook, X, LinkedIn, WhatsApp, Messenger, and copy link
* View each campaign’s reward amount, requirements, geographic eligibility, and status
* See completed, pending, rejected, and paid rewards
* Request a payout once they reach a $100 available balance
* Optionally continue earning until $500 or more before cashing out
* View their profile, payment methods, tax/compliance notices, and activity history

### Reward model

Default reward: **$0.25 per verified eligible share action**, subject to campaign rules and fraud controls.

Do not automatically pay simply because a user clicks a “shared” button. Instead, create a share event and verify it using approved methods such as:

* Native share completion signal when available
* Unique tracked URL creation
* Verified click or conversion thresholds defined by the campaign
* Platform-supported social sharing integrations where permitted
* Manual review for suspicious activity
* Anti-fraud scoring

The UI should clearly label:

* “Share recorded”
* “Pending verification”
* “Eligible reward”
* “Reward approved”
* “Reward rejected”
* “Paid”

# Required five-site promotion gate

Before a user or business can activate earning or campaign features, show a required onboarding step called:

## “Activate Your ShareTree Network”

The user must review and complete sharing tasks for these five platform properties:

1. EARNTREE.co
2. Buyshare.co
3. Bluetruck.co
4. pocketcash.co
5. Burncall.co

Important implementation requirement:

* Do not falsely claim that a post was publicly shared if the platform cannot verify it.
* For each property, create a unique tracking link and let the user choose a supported share method.
* Record the share attempt, link creation, share channel, timestamp, and resulting clicks.
* Show the task as “completed” only when the selected verification requirement is satisfied.
* Include an admin-configurable setting for whether this is mandatory, optional, or replaced by another onboarding requirement.

Activation screen should include:

* Five branded cards
* Description for each property
* Unique tracking link
* Share buttons
* Completion progress bar: 0 of 5 completed
* “Continue” button unlocked after all required tasks are completed
* Clear disclosure that rewards depend on eligibility and verification

# AI matching engine

Create an AI-assisted matching system that recommends campaigns to users based on:

* User location
* User selected service area
* Business campaign location
* Business service radius
* Campaign category
* User interests
* Past campaign engagement
* Preferred share channels
* Campaign budget availability
* Campaign quality score
* Frequency caps
* Fraud risk score
* Local relevance

The first version should use a transparent rules-based matching engine with an architecture that can later support AI ranking.

Example scoring logic:

* Strong location match: high score
* Same business category as user interest: medium score
* Campaign with available budget: required
* Repeated exposure to same campaign: lower score
* Suspicious user activity: exclude or hold for review
* Nearby local businesses: prioritize

Display a “Why this was recommended” label, for example:

* “Recommended because it serves Avon Park and nearby areas.”
* “Matches your interests in food and local services.”
* “Popular with users in your area.”

* Stripe-connected payout account
* Bank account
* Debit card
* PayPal
* Cash App

Important:

Payout rules:

* Hold rewards until verification is complete
* Create payout ledger records

* Include fraud review and account hold capability

# Anti-fraud, privacy, and compliance requirements

Build these controls from the beginning:

## Fraud prevention

* Unique tracked links per user and campaign
* Rate limits on share attempts
* Duplicate click detection
* Bot and VPN risk checks
* Device and session risk signals
* Click quality scoring
* Self-click and repeated-click detection
* IP and device anomaly detection
* Manual review queue
* Admin ability to freeze earnings
* Admin ability to reject or reverse rewards
* Campaign budget protection
* Daily and weekly user earning caps configurable by admin

## Privacy

* Ask for location permission clearly and allow manual location entry
* Do not access a user’s contacts, friends, private messages, or social accounts without explicit consent and a supported integration
* Do not claim to know whether a share reached “real people” unless verified by permitted, reliable signals
* Provide privacy policy, terms of service, reward terms, cookie policy, and consent screens
* Store only necessary personal data
* Encrypt sensitive data
* Never store raw bank account or card numbers

## Legal / disclosure UI

Add placeholders and routes for:

* Terms of Service
* Privacy Policy
* Reward Program Terms
* Advertiser Terms
* FTC endorsement disclosure guidance
* Earnings disclaimer
* Tax information notice
* Prohibited activity policy
* Dispute and appeal process

Include a user-facing disclosure near each campaign:
“Only share content you genuinely support. Add any legally required disclosure when sharing sponsored or rewarded content. Rewards are subject to verification and program rules.”

# Recommended technology stack

Use:

* Frontend: React + TypeScript + Tailwind CSS
* Framework: Next.js or Replit-supported full-stack React architecture
* Backend: Node.js + TypeScript
* Database: PostgreSQL
* ORM: Prisma
* Authentication: Google OAuth with secure session management
* Payments: Stripe Billing + Stripe Connect
* Analytics: first-party event tracking stored in PostgreSQL, designed to later integrate with PostHog or Mixpanel
* File uploads: secure object storage
* Email: Resend or SendGrid
* Maps / geolocation: Google Maps API or Mapbox
* Background jobs: queue system for click processing, fraud checks, reward verification, and payout workflows
* Deployment: Replit deployment-ready configuration
* Environment variables: use secure secrets, never hardcode keys

# Critical build rules

* Do not hardcode payment keys, OAuth secrets, or API keys.
* Use environment variables and include an `.env.example`.
* Add seed data for users, businesses, campaigns, categories, and analytics.
* Build responsive layouts for phone, tablet, and desktop.
* Use realistic mock data where integrations are not configured.
* Add empty states, loading states, validation, error messages, and success messages.
* Make every dashboard action traceable in the database.
* Clearly distinguish mock/demo mode from live payment mode.
* Include README setup instructions.
* Include a checklist of required API keys and integrations.
* Include test accounts and demo seed instructions.
* Build the system so the reward amount, subscription price, payout threshold, and activation requirements can all be changed by admins without code changes.

Start by generating the complete working MVP with authentication, database schema, user and business dashboards, campaign creation, local matching, unique tracking links, share event tracking, Stripe subscription checkout, and an admin dashboard. Use mocked payout processing until Stripe Connect is configured.
